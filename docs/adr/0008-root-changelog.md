# 0008. Maintain a root CHANGELOG alongside per-plugin CHANGELOGs

## Status

Accepted

## Context

Until now the repo kept **only** per-plugin changelogs
(`plugins/<name>/CHANGELOG.md`), each written by `ci/release.py` from that plugin's
Conventional-Commit history. That fits the release mechanism — versions are bumped and
tagged per plugin — but it serves a contributor poorly: there is no single place to read
"what changed in the marketplace as a whole." A reader landing on the public repo has to
open twelve files to reconstruct a release.

A public project benefits from one aggregated, human-curated changelog at the root, in
the widely recognized [Keep a Changelog](https://keepachangelog.com/) shape, that
summarizes program-level changes (cross-plugin work, repo tooling, governance) and points
at the per-plugin files for detail.

This reverses the earlier "per-plugin CHANGELOGs only; no top-level CHANGELOG" working
assumption.

## Decision

Maintain **both**:

- A root `CHANGELOG.md` in Keep a Changelog 1.1.0 format — hand-curated, program-level,
  with an `## [Unreleased]` section that contributors append to in their PRs.
- The existing per-plugin `plugins/<name>/CHANGELOG.md` files — owned by
  `ci/release.py`, the canonical per-plugin release notes keyed to each plugin's tag.

The root changelog is **not** generated by `ci/release.py`; it is curated. The
per-plugin changelogs remain machine-written and authoritative for per-plugin versions.
The two are complementary, not duplicative: the root file aggregates and contextualizes,
the per-plugin files carry the precise per-version record.

## Consequences

**Positive**

- A reader gets the whole-marketplace story from one file.
- Governance and repo-tooling changes (CI, templates, ADRs) now have a home — they
  belong to no single plugin and previously went unrecorded in any changelog.
- The `## [Unreleased]` section gives contributors a clear place to note changes in a PR.

**Negative / mitigations**

- Two changelog surfaces to keep honest. Mitigation: the PR template's checklist names
  both, and the root file explicitly delegates per-plugin detail to the per-plugin files,
  so the root entries stay short (program-level summaries, not per-commit logs).
- The root file is curated, so it can drift if a contributor forgets it. This is a
  warn-by-convention obligation (PR checklist), not a hard gate — consistent with the
  repo's preference for keeping `ci/release.py` the only machine writer of changelogs.
